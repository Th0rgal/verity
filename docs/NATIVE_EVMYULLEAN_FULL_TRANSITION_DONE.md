# Full Native EVMYulLean Transition: Definition Of Done

This document defines what it means to finish the transition from Verity's
custom Yul interpreter to native EVMYulLean execution as the authoritative
compiler-verification target.

It is stricter than the current storage-semantics milestone. The SimpleStorage
native theorem now discharges its retrieve, store, and selector bridge cases,
but the generic compiler theorem stack still uses Verity's custom fuel-based
Yul interpreter through:

```lean
interpretYulRuntimeWithBackend .evmYulLean
```

That path proves the custom interpreter agrees with EVMYulLean-backed builtin
semantics for the bridged fragment. The final transition is complete only when
the public compiler-correctness theorem targets native EVMYulLean execution
directly, or an equivalent wrapper whose only execution engine is
`EvmYul.Yul.callDispatcher`.

## Current Baseline

The current baseline is:

- Layer 2 proves semantic preservation from supported `CompilationModel`
  programs to IR.
- Layer 3 proves IR-to-Yul preservation over Verity's custom statement
  interpreter, retargeted to the `.evmYulLean` backend for builtin execution.
- `Compiler.Proofs.YulGeneration.Backends.Native.interpretRuntimeNative` and
  `interpretIRRuntimeNative` execute through native EVMYulLean, but the generic
  native-vs-interpreter agreement is still an explicit theorem seam.
- `simpleStorage_endToEnd_native_evmYulLean` is a concrete native theorem with
  no public retrieve-hit, store-hit, or selector-miss bridge premises.
- The active compiler stack has zero project-level Lean axioms and zero `sorry`
  placeholders.

## Final Architecture

The completed proof path must be:

```text
CompilationModel
  -> IRContract
  -> emitted Verity Yul runtime
  -> EVMYulLean YulContract
  -> EvmYul.Yul.callDispatcher
  -> projected observable result
  -> public end-to-end theorem
```

The custom Yul interpreter may remain temporarily as a regression oracle, but it
must not be on the proof path for the public semantic-preservation claim.

## Non-Negotiable Completion Criteria

The transition is done only when all criteria in this section are true on
`main`.

### 1. Public Theorem Target

- The generic Layer 3 and end-to-end compiler-correctness theorem statements
  target native EVMYulLean execution, not
  `interpretYulRuntimeWithBackend .evmYulLean`.
- The theorem-facing native target is either `Native.interpretIRRuntimeNative`
  or a narrower wrapper that internally lowers to EVMYulLean and executes via
  `EvmYul.Yul.callDispatcher`.
- No public theorem keeps a generic native-vs-custom-interpreter agreement
  premise such as `nativeIRRuntimeAgreesWithInterpreter`,
  `nativeCallDispatcherAgreesWithInterpreter`, or an equivalent bridge
  assumption.
- Any remaining comparison with the custom interpreter is explicitly marked as
  regression or differential testing, not as the authoritative proof target.

### 2. Generated Fragment Characterization

- The compiler-emitted Yul fragment is represented by named syntactic
  predicates or theorem hypotheses that can be generated from the compiler
  output without manual contract-specific proof work.
- The characterization covers generated statements, expressions, helper
  functions, dispatcher shape, storage layout, memory/return ABI snippets,
  event/log snippets, selector extraction, callvalue guards, calldata guards,
  and native switch temporaries.
- Unsupported constructs fail closed before the public theorem claims semantic
  preservation. They must not silently fall back to custom interpreter
  semantics.
- The generated-fragment contract is tested by CI so new emitted Yul forms
  cannot bypass the native proof surface.

### 3. Native State Bridge

- Every transaction, environment, and state field reachable from generated Yul
  has one of three statuses:
  - bridged with a theorem,
  - explicitly constrained by a theorem-facing invariant, or
  - rejected fail-closed before execution.
- The bridge covers selector/calldata bytes, calldata word reads, caller,
  source, current address, callvalue, timestamp, block number, supported chain
  and blob fields, storage, transient storage if emitted, memory, returndata,
  logs, and rollback behavior.
- State projection uses bounded EVM-compatible storage keys and words. No proof
  relies on Nat slot injectivity after coercion to 256-bit EVM keys.
- Any EVMYulLean fork divergence required by the state bridge is either
  upstreamed or documented in the fork audit with a non-semantic explanation.

### 4. Native Statement And Expression Semantics

- For every statement and expression form generated by the compiler, there is a
  theorem proving native EVMYulLean execution agrees with the source/IR
  semantics used by the compiler proof.
- Coverage includes literals, locals, assignments, blocks, conditionals,
  switches, loops if emitted, function/helper calls, arithmetic, comparisons,
  bit operations, calldata operations, storage operations, memory operations,
  mapping-slot helpers, returns, stops, reverts, and logs.
- Dispatcher execution is proved for selector hits, selector misses, default
  arms, duplicate-case fail-closed behavior, matched-flag preservation, and
  generated native switch temporary freshness.
- External calls, static calls, returndata, and gas-sensitive behavior are
  either fully modeled and proved or syntactically excluded from the generated
  native fragment used by the public theorem.

### 5. Result Projection

- Native success, stop, return, revert, and hard-error results are projected to
  the theorem-facing observable result with proved lemmas.
- Projection covers success flags, return values, final storage on observable
  slots, final mappings if still present in the result shape, event/log
  histories, rollback semantics, and missing-account/missing-slot defaults.
- Non-word-sized return or revert buffers are either modeled faithfully or
  excluded by a generated-fragment theorem.
- Projection lemmas are not contract-specific; concrete contract theorems may
  instantiate them, but the generic transition cannot depend on SimpleStorage
  closed forms.

### 6. Fuel And Termination Story

- Fuel cannot appear as an unexplained public assumption.
- The public theorem uses one of these designs:
  - a fuel-parametric theorem with explicit monotonicity/sufficiency facts,
  - a generated-code fuel lower bound proved from the emitted runtime, or
  - a total wrapper that chooses and proves a sufficient fuel internally.
- Fuel exhaustion must project as a closed failure only when the theorem
  explicitly reasons about that failure. It must not be confused with semantic
  disagreement.

### 7. Proof Cleanliness

- No `sorry`, `admit`, `axiom`, or theorem declaration marked as an assumption
  is introduced for the transition.
- `AXIOMS.md`, generated axiom reports, and `PrintAxioms.lean` agree that the
  active compiler stack has zero project-level Lean axioms.
- The public native theorem's `#print axioms` output contains only Lean
  foundational axioms such as `propext`, `Classical.choice`,
  `Lean.ofReduceBool`, or `Quot.sound`.
- There are no hidden local assumptions that reintroduce custom interpreter
  correctness, Nat storage-slot injectivity, unbounded storage values, or
  unconstrained environment defaults.

### 8. Documentation And Trust Boundary

- `README.md`, `AUDIT.md`, `TRUST_ASSUMPTIONS.md`,
  `docs/NATIVE_EVMYULLEAN_TRANSITION.md`, and
  `docs/VERIFICATION_STATUS.md` describe native EVMYulLean as the
  authoritative semantic target only after the theorem target has flipped.
- Legacy custom-interpreter files are documented as reference-oracle or
  regression infrastructure.
- EVMYulLean fork audit artifacts and capability reports are regenerated and
  checked.
- The transition docs name all remaining excluded features and link them to
  issue-tracked scope decisions.

### 9. CI Enforcement

- CI fails if the public compiler theorem drifts back to the custom
  interpreter path.
- CI checks the generated-fragment boundary, native transition docs, bridge
  coverage reports, fork audit artifacts, axiom report, proof-length metadata,
  and native smoke/regression tests.
- The following local validations pass before merge:

```bash
lake build Compiler.Proofs.YulGeneration.Backends.EvmYulLeanNativeHarness
lake build Compiler.Proofs.YulGeneration.Backends.EvmYulLeanRetarget
lake build Compiler.Proofs.EndToEnd
lake build Compiler
lake build Contracts
lake build PrintAxioms
python3 scripts/generate_print_axioms.py --check
python3 scripts/check_native_transition_doc.py
python3 scripts/check_bridge_coverage_sync.py
python3 scripts/check_evmyullean_capability_boundary.py
make check
```

### 10. Merge Readiness

- Every PR in the transition stack is based on `main` or the PR directly below
  it.
- Required-equivalent GitHub checks are green on the head commit.
- Cursor Bugbot, Codex, or maintainer review threads on the head commit are
  resolved.
- Generated files are produced by scripts, not hand-edited.
- Branches are pushed with `git push --force-with-lease` when rebasing is
  necessary; plain `--force` is never used.

## Incremental PR Shape

Each PR should move exactly one durable boundary:

1. Native generated-fragment predicate or checker.
2. Native state/environment bridge theorem family.
3. Native result projection theorem family.
4. Native statement/expression equivalence for one generated construct family.
5. Dispatcher/native-switch preservation.
6. Fuel sufficiency or fuel-parametric theorem layer.
7. Generic native-vs-interpreter agreement for generated Yul.
8. Public theorem target flip.
9. CI/doc/trust-boundary cleanup after the flip.
10. Legacy interpreter demotion or removal.

Do not mix proof work with unrelated cleanup. Do not remove the custom
interpreter until the public theorem target has flipped and at least one PR has
kept it only as regression infrastructure.
